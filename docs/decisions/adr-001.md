# ADR-001: Use Helm as the Application Packaging and Templating Tool

**Status:** Accepted

---

## Context

Kubernetes manifests (Deployments, Services, Ingress, ConfigMaps, etc.) are verbose YAML files. As the number of applications and environments grows, managing raw manifests becomes repetitive and error-prone — the same manifest structure is duplicated across environments with only small differences (image tag, replica count, resource limits). A templating or packaging tool is needed to manage this complexity in a maintainable and reusable way.

Three approaches were considered: raw Kubernetes manifests, Kustomize, and Helm.

---

## Decision

Use Helm as the application packaging and templating tool for all workloads defined in `platform-charts`.

---

## Reasoning

### Raw manifests

Raw YAML manifests are the simplest starting point but do not scale. Environment-specific differences require either duplicating entire manifests per environment (high maintenance burden) or using `sed`/`envsubst` substitutions (fragile, not version controlled). Raw manifests have no concept of packaging, versioning, or reuse.

### Kustomize

Kustomize is a template-free configuration management tool built into `kubectl`. It works by layering patches on top of a base manifest — a `base/` directory holds the canonical manifest, and `overlays/` directories patch environment-specific values on top. Kustomize is a good fit when the differences between environments are small and structural (e.g. adding a label, changing a replica count).

Kustomize's strength is also its limitation: it is patch-based rather than parameterised. When differences between environments are larger (different ingress hostnames, different resource profiles, different secrets backends), Kustomize patches become verbose and hard to reason about. It also has no native concept of packaging or versioning — charts cannot be published, versioned, or consumed across teams.

### Helm

Helm is the de facto standard for packaging, versioning, and deploying Kubernetes applications. A Helm chart is a parameterised template — values are injected at render time, making it straightforward to produce different manifests for different environments from a single chart definition. Charts are versioned artifacts that can be published to a registry (Harbor in this case) and consumed by ArgoCD.

Helm's templating model is a natural fit for `platform-charts` where the goal is a reusable chart library consumed by multiple applications. A single `ml-service` chart can serve `ml-classifier` today and any future ML service with different values — Kustomize cannot replicate this without duplicating base manifests per application.

Helm also has first-class ArgoCD integration — ArgoCD natively renders Helm charts and injects values from `platform-deploy`, completing the GitOps loop cleanly.

---

## Alternatives Considered

| Tool | Reason Not Chosen |
|---|---|
| Raw Kubernetes manifests | No templating, no reuse, high duplication across environments |
| Kustomize | Patch-based model becomes verbose for larger environment differences; no native packaging or versioning; less suitable for a shared chart library |
| Jsonnet / CUE | Powerful but steep learning curve; small ecosystem relative to Helm; not worth the overhead for this use case |

---

## Consequences

**Positive:**
- De facto standard — widely used in production environments and directly relevant to ML Platform Engineer roles
- Parameterised templating enables a single chart to serve multiple applications and environments
- Native ArgoCD integration — Helm values injected from `platform-deploy` at sync time
- Charts are versioned artifacts — publishable to Harbor and consumable across teams
- Large ecosystem — extensive community charts available as reference

**Negative:**
- Helm's Go templating syntax can become complex and hard to read for deeply nested values
- Chart debugging requires rendering templates locally (`helm template`) — adds a step to the development workflow
- Helm manages release state in Kubernetes Secrets — an additional layer of state to be aware of alongside ArgoCD's own state
