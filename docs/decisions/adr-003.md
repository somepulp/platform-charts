# ADR-003: python-service Helm Chart Design

**Status:** Accepted

---

## Context

The `python-service` chart is the first chart in `platform-charts` and the deployment vehicle for `ml-classifier`. Decisions are needed on what Kubernetes resources the chart manages, how it is structured, and what platform standards it enforces by default. The chart must be generic enough to serve any Python web service while being opinionated enough to enforce platform standards without requiring application teams to rediscover them.

---

## Decision

The `python-service` chart manages the following Kubernetes resources:

- `Deployment` — runs the containerized Python service
- `Service` — exposes the Deployment internally within the cluster
- `HTTPRoute` — Gateway API route that exposes the service externally via Envoy Gateway
- `HorizontalPodAutoscaler` — scales replicas based on CPU/memory utilization

The chart enforces the following platform standards by default:

- Resource requests and limits required — no unbounded workloads
- Liveness and readiness probes required — no unobservable workloads
- Prometheus scrape annotations on the pod — all workloads are observable by default
- Non-root container security context — no unnecessary privilege escalation

---

## Reasoning

### Resources included

**Deployment** is the core resource — it manages the pod lifecycle, rolling updates, and replica count.

**Service** is required for the HTTPRoute to have a backend to route traffic to. It also enables service discovery within the cluster if other services need to call this one internally.

**HTTPRoute** is the Gateway API resource that exposes the service externally. Using HTTPRoute rather than the legacy Ingress resource aligns with the Gateway API adoption decision in `platform-components` ADR-002. HTTPRoute is toggled off by default (`httpRoute.enabled: false`) so the chart can be used for internal-only services without modification.

**HorizontalPodAutoscaler** is included because ML inference services have variable load patterns — request volume can spike significantly. HPA is toggled off by default (`hpa.enabled: false`) and enabled per application in `platform-deploy` values.

### Resources excluded

**PersistentVolumeClaim** — model weights are baked into the container image for `ml-classifier`. If a future service requires persistent storage, this can be added to the chart at that point (the natural trigger for chart evolution).

**ConfigMap / Secret** — application configuration is injected via environment variables sourced from Vault. The chart supports `env` and `envFrom` value keys to pass these through without owning the ConfigMap or Secret lifecycle.

### Platform standards enforced by default

Resource requests and limits are required rather than optional. A chart that allows unbounded resource consumption would undermine the platform's ability to schedule workloads reliably on the Pi. Sensible defaults are provided but must be explicitly overridden per application.

Liveness and readiness probes are required. Without them, Kubernetes cannot distinguish a healthy pod from a deadlocked one, and rolling updates cannot proceed safely. The chart defaults to an HTTP probe on `/health` — FastAPI applications expose this endpoint by convention.

Prometheus scrape annotations are added to every pod by default. This ensures all workloads are scraped by Prometheus without requiring application teams to remember to add annotations. Observability is opt-out, not opt-in.

A non-root security context is set by default (`runAsNonRoot: true`, `allowPrivilegeEscalation: false`). This is a baseline security posture that should apply to all application workloads.

## Chart Structure

```
python-service/
  Chart.yaml          # chart metadata and version
  values.yaml         # defaults for all configurable values
  templates/
    deployment.yaml   # Deployment resource
    service.yaml      # ClusterIP Service resource
    httproute.yaml    # Gateway API HTTPRoute (toggled by httpRoute.enabled)
    hpa.yaml          # HorizontalPodAutoscaler (toggled by hpa.enabled)
    _helpers.tpl      # shared template helpers (labels, selectors, names)
```

## Key values.yaml Structure

```yaml
image:
  repository: ""        # required — no default
  tag: "latest"
  pullPolicy: IfNotPresent

replicas: 1

resources:
  requests:
    cpu: 100m
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi

probes:
  liveness:
    path: /health
    initialDelaySeconds: 10
  readiness:
    path: /health
    initialDelaySeconds: 5

httpRoute:
  enabled: false
  hostname: ""
  gatewayName: ""
  gatewayNamespace: ""

hpa:
  enabled: false
  minReplicas: 1
  maxReplicas: 3
  targetCPUUtilizationPercentage: 80

securityContext:
  runAsNonRoot: true
  allowPrivilegeEscalation: false

metrics:
  enabled: true         # Prometheus scrape annotations
  port: 8000
  path: /metrics

env: []                 # environment variables passed through to the container
envFrom: []             # references to ConfigMaps or Secrets for env vars
```

## Vault Integration Approach

Two patterns exist for integrating HashiCorp Vault with Kubernetes workloads:

**Vault Agent Injector** — a mutating webhook injects a sidecar container into the pod. The sidecar authenticates to Vault, fetches secrets, and writes them to a shared volume. Requires Vault-specific annotations on the pod in `deployment.yaml`.

**Vault Secrets Operator (VSO)** — Vault syncs secrets into native Kubernetes `Secret` objects via a `VaultStaticSecret` or `VaultDynamicSecret` CRD. The pod consumes the resulting Kubernetes Secret via `envFrom` — it never interacts with Vault directly.

**VSO is chosen** for the following reasons:

- The `python-service` chart remains Vault-agnostic — no Vault-specific annotations or sidecar configuration in `deployment.yaml`
- Secrets arrive as standard Kubernetes Secrets, consumed via the existing `envFrom` passthrough in `values.yaml`
- The `VaultStaticSecret` CRD that instructs VSO which secrets to sync lives in `platform-deploy` per application — keeping secret mapping close to the deployment manifest that needs it
- VSO is HashiCorp's current recommended pattern for Kubernetes secret consumption

This means no additional Vault-specific template is needed in `python-service`. The `envFrom: []` value key is sufficient for applications to reference VSO-synced secrets.

---

## Consequences

**Positive:**
- Enforces resource limits, probes, observability, and security context as platform defaults — application teams cannot accidentally skip them
- HTTPRoute and HPA are opt-in via feature flags — chart works for simple internal services without modification
- `env` and `envFrom` passthrough gives application teams flexibility for configuration without the chart owning secrets lifecycle
- `_helpers.tpl` centralizes label and selector logic — consistent labeling across all resources

**Negative:**
- Opinionated defaults may require explicit overrides for edge cases (e.g. a service that legitimately needs to run as root)
- As a single chart serving all Python services, it will eventually accumulate feature flags — the trigger for migration to library chart inheritance (see ADR-002)
- `/health` probe path is a convention, not a standard — application code must implement it
