# ADR-002: Shared Chart Library Structure, Naming Convention, and Deferred Library Chart Inheritance

**Status:** Accepted

---

## Context

When packaging application workloads with Helm, several structural decisions are needed:

1. Should charts live in a centralized shared library or co-located with application source code?
2. How should charts be named and categorized — by workload type (e.g. `ml-service`) or by runtime (e.g. `python-service`)?
3. Should Helm library chart inheritance be implemented from the start, or deferred until the complexity is warranted?

---

## Decision

1. Maintain all application Helm charts in the centralized `platform-charts` repository
2. Name charts by runtime (`python-service`) rather than workload type (`ml-service`)
3. Implement regular deployable charts now; defer Helm library chart inheritance until a second application exposes real divergence that motivates it

---

## Reasoning

### 1. Shared chart library over per-app charts

Co-locating charts with application source code works for single-app projects but does not scale. As applications grow, each reinvents the same chart structure — Deployment, Service, HTTPRoute, HPA, resource limits, health checks, observability instrumentation — with no enforcement of platform standards and no single place to apply cross-cutting changes.

A centralized chart library inverts this: the platform team owns chart definitions and enforces deployment standards, while application teams supply only workload-specific values (image, replicas, resource limits, ingress hostname). This mirrors the organizational boundary between platform engineers (who own `platform-charts`) and application/ML engineers (who own source code) — directly relevant to ML Platform Engineer roles.

### 2. Runtime-based naming over workload-type naming

Naming the chart `ml-service` assumes all ML workloads share the same deployment pattern — a significant assumption. A batch training job looks nothing like a FastAPI inference server. Workload-type naming also does not generalize: if a non-ML Python service is added later, `ml-service` is the wrong chart for it.

Naming by runtime (`python-service`) is more honest and more reusable:
- Any Python web service (ML or otherwise) can consume `python-service`
- The chart describes *how* to deploy a Python service, not *what* it does
- Additional runtimes (if ever needed) follow the same pattern: `node-service`, `java-service`

### 3. Deferred library chart inheritance

Helm library charts (`type: library` in `Chart.yaml`) define reusable named template helpers but produce no manifests on their own. They are consumed by application charts via a `dependencies` block, enabling full template inheritance. This is the most mature pattern for a shared chart library serving multiple teams.

However, for a single application (`ml-classifier`), library chart inheritance adds two charts to maintain (the library and the app chart) before the second application exists to justify the abstraction. The complexity cost is not warranted yet.

The simpler pattern — a single deployable `python-service` chart consumed directly by `platform-deploy` — is the right starting point. The migration path to library chart inheritance is well-defined and mostly mechanical when the time comes (see Future Evolution below).

---

## Resulting Structure

```
platform-charts/
  charts/
    python-service/
      Chart.yaml
      values.yaml
      templates/
        deployment.yaml
        service.yaml
        httproute.yaml
        hpa.yaml
```

`platform-deploy` references `python-service` directly and supplies application-specific values per environment branch.

---

## Future Evolution: Migration to Library Chart Inheritance

The natural trigger for migration is when a second Python application has deployment requirements that cannot be cleanly accommodated by `python-service` values alone — for example, a sidecar container, a background worker, or a fundamentally different health check pattern. When the chart starts accumulating conditionals to handle multiple application types, it is time to refactor.

The migration is mechanical:

1. Convert `python-service` to a library chart (`type: library` in `Chart.yaml`)
2. Wrap each template in a named template definition (`{{- define "python-service.deployment" -}}`)
3. Create thin application charts (`apps/ml-classifier/`) that declare `python-service` as a dependency and call its named templates
4. Update `platform-deploy` to reference the application chart instead of `python-service` directly — values structure is largely unchanged

Deferring this migration until real divergence is observed means the library chart is built around actual requirements rather than hypothetical ones. It also provides a concrete refactoring experience that reinforces understanding of why library charts exist.

---

## Alternatives Considered

| Approach | Reason Not Chosen |
|---|---|
| Per-app charts co-located with source code | No reuse, no platform standards enforcement, duplicated chart logic as app count grows |
| Mono-repo with all charts and source code | Conflates platform and application concerns |
| `ml-service` chart name | Overly specific to ML workloads; does not generalize to other Python services |
| Library chart inheritance from day one | Adds two-chart complexity for a single application; premature abstraction before real divergence is observed |

---

## Consequences

**Positive:**
- Single place to enforce platform-wide deployment standards
- Runtime-based naming is honest and generalizes as applications grow
- Simple deployable chart reduces day-one complexity
- Documented migration path to library charts means the evolution is planned, not reactive
- The refactoring experience itself reinforces understanding of Helm's abstraction model

**Negative:**
- `python-service` chart will eventually accumulate conditionals as applications diverge — the trigger for migration to library charts
- Chart and application code in separate repos — iterating on both simultaneously requires two PRs
- Migration to library charts requires coordinated changes across `platform-charts` and `platform-deploy`
